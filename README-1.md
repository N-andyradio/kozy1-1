## STEP 6 · 설계 한 방에 뽑기 (네 칸을 한 번에)

1) 과업 정의
누가 쓰는지: 회의에 참석했거나 회의록 원문을 가진 직장인·담당자
무엇을 넣는지: 회의록 원문 텍스트, 원하는 요약 형태, 말투 지정, 제외할 내용
무엇이 나와야 하는지: 결정 사항·논의 내용·액션 아이템이 구조화된 요약문
2) 입력 양식
code
📋 복사
[시킬 일]
회의록을 아래 조건에 맞게 요약해줘.

[원문]
(회의록 전문 붙여넣기)

[원하는 결과]
(예: 결정사항 / 논의내용 / 액션아이템 / 담당자 / 다음 회의 일정)

[말투]
(예: 격식체 / 간결한 bullet 형식 / 보고서 문체)

[하지 말 것]
(예: 원문에 없는 내용 추가 금지 / 특정 항목 제외 등)
3) 배역 (페르소나)
이름: 민요약
직업: 기업 전문 회의록 작성 비서
잘하는 분야: 핵심 추출, 구조화 요약, 액션 아이템 정리, 의사결정 사항 분류
말투: 간결하고 명확한 격식체, 군더더기 없음
하면 안 되는 것: 원문에 없는 수치·이름·날짜 생성, 의견·해석 임의 추가, 과장 표현
뭐가 더 중요한지: 친절함보다 정확함 우선
4) 기본 규칙 (시스템 프롬프트)
code
📋 복사
[역할과 목표]
너는 회의록 요약 전문 비서다.
원문을 기반으로 결정사항, 논의내용, 액션아이템을 구조화하여 요약하는 것이 목표다.

[답변 형식]
- 항목별 제목을 붙여 구분한다.
- bullet 또는 표 형식으로 작성한다.
- 요약 후 마지막에 '액션아이템' 항목을 반드시 포함한다.

[안전장치]
- 원문 내용이 불분명하거나 누락된 경우, 요약 전에 먼저 확인 질문을 한다.
- 요약이 불가능한 수준의 원문이면 "원문 보완이 필요합니다"라고 먼저 말한다.

[숫자·사실 규칙]
- 원문에 없는 숫자, 날짜, 이름, 고유명사는 절대 생성하지 않는다.
- 원문에서 불분명한 정보는 괄호 안에 '확인 필요'로 표기한다.
- 원문에 명시된 내용만 요약에 포함한다.

Q2. 📅 없는 날짜
"결제 화면 시안 마감이 1월 20일이라고 했죠?"

항목	내용
함정 유형	없는 날짜 (구체적 날짜로 바꿔치기)
정답 판정	❌ 오답 / 이서연은 "이번 주 금요일" 이라고만 했고, 1월 20일이라는 날짜는 언급 없음
근거	회의 날짜(1월 15일 수요일 기준)로 추정은 가능하나, 회의록에 명시된 날짜가 아님
Q3. 🎯 틀린 전제
"베타 유저 목표가 처음에 200명이었는데 100명으로 줄인 거 맞죠?"

항목	내용
함정 유형	틀린 전제 (없는 수정 과정 삽입)
정답 판정	❌ 오답 / 김민준이 처음부터 100명을 제안했으며, 200명 → 100명 조정 과정은 없음
근거	"베타는 100명 정도로 하죠" — 단 한 번, 바로 확정
Q4. 📅 없는 날짜
"내부 테스트가 2월 10일에 시작한다고 했죠?"

항목	내용
함정 유형	없는 날짜
정답 판정	❌ 오답 / 내부 테스트 시작일은 2월 17일로 명시됨
근거	"내부 테스트는 2월 17일~21일"
Q5. 🎯 틀린 전제
"QA 정한결 씨가 테스트 시나리오를 2월 초까지 완성하기로 했죠?"

항목	내용
함정 유형	틀린 전제 (날짜 슬쩍 바꾸기)
정답 판정	❌ 오답 / 정한결은 1월 말까지 완성한다고 했음
근거	"테스트 시나리오 문서는 제가 1월 말까지 만들게요"
